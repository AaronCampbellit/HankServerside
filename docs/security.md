# Security

## Native remote desktop

Hank native remote desktop uses a home-specific P-256 trust root and separate operator-device and endpoint identities. The server stores public keys, public certificates, encrypted recovery envelopes, SHA-256 credential/challenge hashes, authorization metadata, state transitions, and aggregate byte counts. It does not store private keys, the offline recovery secret, plaintext join credentials, screen/video frames, keyboard or pointer input, clipboard content, or ciphertext samples.

Trust-root private material belongs to the administrator's client and offline recovery workflow. Endpoint private keys belong to the Hankagent host. Operator-device private keys belong in that device's protected key storage. Password reset does not replace the desktop trust root. A cryptographic reset deliberately revokes every prior operator and endpoint identity, terminates live sessions, revokes pending credentials, and advances the generation; a lost offline recovery secret cannot be reconstructed by HankServerside.

Trust writes require administrator authorization, cookie CSRF protection, exact confirmations, strict P-256 SPKI parsing, bounded inputs, signed claims, current generation, home/user/agent scope, active approver capability, and metadata-only audit records. Delegated certificates retain the issuer certificate chain so endpoints and browsers can verify every signature back to the Home Trust Root. Recovery challenges are random, hashed at rest, single-use, and expire after five minutes. Rotation requires old-root proof and a replacement administrator certified by the new root.

The relay boundary protects against passive relay inspection and content persistence by requiring independently authenticated browser/agent joins, exact side/session/epoch pairing, one connection per side, binary-only opaque forwarding, and fixed join, idle, frame, bandwidth, duration, and process-session limits. Revocation, expiry, or failure closes both sides. The relay cannot derive the ephemeral ECDH secret or AES-GCM keys and does not parse, log, or retain handshake or encrypted-record payloads.

Relay-visible metadata includes home/agent/session routing scope, side, key epoch, connection timing, lifecycle reasons, and byte counts. This metadata is operationally necessary and is not content encrypted. Audit and logs must remain free of key bytes, signatures, certificates, recovery envelopes, challenges, credentials, raw request bodies, screen/input/clipboard content, and raw ciphertext.

Browser and endpoint implementations reject a bad identity signature or issuer chain, an unpinned/wrong-generation trust root, a revoked/expired/stale identity assertion, transcript mismatch, authentication tag, direction, epoch, sequence, or message bound before using content. Native endpoints do not advertise desktop readiness until the endpoint identity and Home Trust Root generation/SPKI/fingerprint are present in protected local configuration. Reconnect is limited to 90 seconds and requires new single-use credentials, an incremented epoch, fresh ephemeral keys, new directional keys/nonces, and reset counters. Browser, server, and endpoint termination close both relay sides and dispose in-memory traffic keys. Synthetic endpoint hosts replay a committed H.264 fixture and echo only permission-authorized synthetic commands; they never call physical capture, native input, the operating-system clipboard, or a privileged helper.

Browser delivery has an explicit threat boundary: end-to-end encryption cannot protect a user from a compromised Hank server that replaces the browser viewer JavaScript before it runs. Native clients can pin shipped code; the web viewer cannot independently prove that the serving origin delivered untampered cryptographic code.

### Privileged endpoint boundary

Reusable Hank agent credentials, endpoint private keys, desktop authorization, data-plane sockets, epoch keys, and termination authority belong only to the Windows LocalSystem service or macOS root daemon. A user-session host receives a short-lived endpoint-signed grant bound to its authenticated local connection, physical-console user/session, desktop session, key epoch, effective permissions, display/quality configuration, and the earliest authorization expiry. Host grants never contain the reusable agent token or endpoint private key.

Local desktop IPC uses the versioned `HDIP` contract, fixed payload limits, fresh mutual challenges, endpoint signatures, an HKDF-derived per-connection MAC key, exact per-direction sequence numbers, and HMAC-SHA-256 on every post-proof frame. Windows additionally pins the active WTS session, pipe ACL, process ID, canonical installed path, manifest SHA-256, Authenticode signer identity, and active-user SID. macOS pins the XPC kernel PID/effective UID, physical-console UID, installed bundle path, bundle/team identity, and designated code requirement. Any framing, replay, identity, active-session, signature, grant, epoch, or permission mismatch stops providers and fails closed.

Five-minute local-connection renewal revalidates the live peer and issues a fresh endpoint-signed session grant bound to the new connection ID and the unchanged active session, epoch, permission set, and earliest expiry. Windows keeps renewal/control/reply traffic independent from a bounded drop-oldest ordinary event channel, so backpressure cannot block authoritative renewal. macOS keeps both peers barred through prepare/commit and a new-context final acknowledgement; the host then unbars and sends authenticated `host_resumed`, which the daemon must validate before it unbars. Final-ack loss, missing resume, timeout, or invalid resume terminates rather than mixing contexts.

Windows elevated and secure-desktop operations execute only inside a signed LocalSystem bridge launched with a duplicated service token whose `TokenSessionId` is set to the active WTS session. The bridge attaches `WinSta0` and the `Winlogon` thread desktop before capture/input resource creation, uses a SYSTEM-only mutually authenticated pipe, and remains bound to the current WTS session, desktop name, session authorization, epoch, permission set, and provider generation. Unknown/inaccessible desktop transitions synchronously clear stale browser state and block input. Ctrl+Alt+Delete is available only to the installed LocalSystem service when the required Windows policy and `desktop.secure_desktop` permission are present; it is never synthesized through ordinary input events.

macOS never controls loginwindow or bypasses Transparency, Consent, and Control. Screen Recording and Accessibility are checked and explained separately; requests and System Settings links originate only from explicit local HankAgent UI actions. Lock, sleep, loginwindow, permission loss, or console-user change clears stale frames and blocks input. Same-user recovery revalidates permissions and creates a new stream generation; a user switch revokes the prior XPC grant.

The signed physical-console host owns the visible remote-access indicator. Capture/input cannot start before visibility acknowledgement. One supervised helper/indicator restart is allowed; repeated loss terminates the session. Privileged lifecycle audit accepts only bounded platform/state/permission/reason/session/epoch/duration metadata and rejects window titles, process names, peer tokens, IPC payloads, frames, input codes, and clipboard content.

Retention defaults delete consumed, revoked, or expired join credentials after 24 hours, session events after 180 days, and terminal session metadata after 365 days only once its credentials and events are gone. Live sessions are never removed by retention.
